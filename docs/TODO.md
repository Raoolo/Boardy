- [ ] **Host su Oracle Cloud Free Tier (Always Free ARM)** — alternativa "spesa 0" al server fisico / Hetzner per esporre Boardy (+ un secondo sito) pubblicamente. Decisione presa il 2026-05-29: si parte da qui perché è gratis per sempre, non solo il primo mese. Chiarimenti chiave da non dimenticare:
  - **Tre livelli Oracle distinti** (fonte dell'equivoco "gratis 1 mese poi paghi"): (1) **Free Trial** = 300$ di credito da spendere in 30gg, *scade*; (2) **Always Free** = risorse gratis *per sempre*, include la ARM Ampere A1 (fino a 4 OCPU + 24GB RAM totali) — **questo è il target**; (3) Pay As You Go = a consumo. Usando solo Always Free si paga **0€ a tempo indeterminato** anche dopo il primo mese.
  - **Carta richiesta** alla registrazione (NO prepagata) solo per verifica identità: pre-auth di ~1€ rimborsata, non è una ricarica. Resta on-file per eventuali sforamenti → impostare **budget alert a 0€/1€** subito.
  - **Catch noti**: (a) `Out of host capacity` frequentissimo sulla ARM — è il motivo dello script sotto; (b) reclaim delle istanze idle su account solo-free (mitigazione: upgrade a PAYG restando dentro i limiti free = niente reclaim, sempre 0€); (c) sforare i limiti = addebito vero sulla carta.
  - **Lavoro tecnico vero**: rebuild dell'immagine Docker per **ARM (`linux/arm64`)** — `deploy/Dockerfile` oggi è implicitamente x86. Attenzione a `torch`/`sentence-transformers` su ARM (il modello e5 carica ~1–2GB in RAM, ma 24GB di Always Free coprono con margine enorme). Cloudflare Tunnel già in `docker-compose.yml` → niente porte/IP statico/TLS da gestire, stesso `--profile tunnel`.
  - **Sub-task: script auto-find capacity ARM** — automatizzare il retry del provisioning contro `Out of host capacity`. Tipicamente loop su OCI CLI / SDK Python (`oci.core.ComputeClient.launch_instance`) con backoff, che ritenta finché una availability domain libera una A1; opzionale notifica (Telegram? riusare il bot) quando l'istanza parte. Da scrivere quando si passa alla fase esecutiva.
  - Vedi anche gli item self-host in Low priority (CF Tunnel, dominio esistente) — il flusso di deploy è lo stesso una volta che la VM ARM è in piedi.

- [ ] **Dev-local ↔ deploy-server convergence** — una volta trasferito lo snapshot iniziale e attivato il deploy, due DB divergono in fretta: io scrivo una conv sul localhost (test/dev), gli amici ne scrivono altre sul deploy, niente merge automatico. Da capire dopo qualche settimana d'uso quale pattern preferire:
  - (a) **Deploy = prod, locale = throwaway**: tutte le interazioni reali avvengono sul server. Il `boardy.db` locale viene rigenerato/wipato come ambiente di test ogni volta che faccio dev. Cambio routine ("non scriverci mai roba che vuoi tenere"). Più pulito modello, ma perdo l'app a portata di mano quando lavoro offline.
  - (b) **Pull on demand**: piccolo script `etl/pull_from_server.py` che fa `scp` (o `docker run --rm -v boardy_boardy_db:/data alpine cat /data/boardy.db > local.db`) del DB dal server al locale prima di una sessione di dev. Push code-only, mai DB. Pro: lavoro offline con dati freschi. Con: ricordarsi di tirare giù la versione, foot-gun se dimentico e finisco a editare un DB locale stale.
  - (c) **Tunnel SSH al DB remoto**: dev locale apre un SSH tunnel e parla al DB live del server. Pro: zero sync. Con: latency su ogni query (SQLite over network non è il suo elemento), write da dev corrompono "prod".
  - (d) **Backup periodico server → cloud / repo privato**: cron sul server che fa `sqlite3 backup` e synca su un bucket/branch. Locale legge da lì. Combina (b) con automation. Effort extra.
  Mia preferenza ipotetica: **(a)** — è il pattern "il prod è il prod". Lo sviluppo nuovo gira sempre contro un DB di test (può anche essere lo snapshot iniziale rinominato). Da decidere dopo qualche settimana di uso reale: quanto spesso scrivo dati "che voglio tenere" sul localhost? Se ~mai, (a) è perfetta. Se spesso, serve (b)/(d).
